# Investigate, verify and fix issue due to failing to clear historical data across context changes (like switching cars, or teleporting from the garage to the track). #379

**Open**
**coasting-nc** opened this issue on Mar 16, 2026

## Description

Investigate, verify and fix issue due to failing to clear historical data across context changes (like switching cars, or teleporting from the garage to the track).

## Question

Review the whole codebase. Are there any other things that we should reset or update or something like that, when we change cars? Are there any other similar bugs or issues due to not updating the internal state or logic when changing cars?

## Answers

This is a fantastic question. When dealing with a stateful physics engine running at 400Hz, failing to clear historical data across context changes (like switching cars, or teleporting from the garage to the track) is the #1 cause of math explosions, NaN propagation, and violent FFB jolts.

I have reviewed the entire codebase looking for variables that track "history" or "previous state". Here are the 4 hidden bugs related to state carry-over, and exactly how to fix them.

---

### Bug 1: Slope Detection Buffers (Cross-Car Contamination)

**The Issue:**
The Slope Detection algorithm uses 41-sample circular buffers (`m_slope_lat_g_buffer`, `m_slope_slip_buffer`, etc.) to calculate the Savitzky-Golay derivatives.
Currently, these buffers are only cleared when the user toggles the feature on/off in the GUI. If you switch cars, the buffer contains 20 samples from the old car and 21 samples from the new car. The derivative calculation across that boundary will result in an astronomical spike, instantly dropping your grip factor to 0.2 for the first few seconds of driving.

**The Fix:**
In `src/physics/GripLoadEstimation.cpp`, inside `InitializeLoadReference` (which we established runs on every car change), clear the slope state:

```cpp
void FFBEngine::InitializeLoadReference(const char* className, const char* vehicleName) {
    std::lock_guard<std::recursive_mutex> lock(g_engine_mutex);
    ResetNormalization();

    // ... (the missing telemetry resets we added earlier) ...

    // --- NEW FIX: Clear Slope Detection History ---
    m_slope_buffer_count = 0;
    m_slope_buffer_index = 0;
    m_slope_smoothed_output = 1.0;
    m_slope_lat_g_smoothed = 0.0;
    m_slope_slip_smoothed = 0.0;
    m_slope_torque_smoothed = 0.0;
    m_slope_steer_smoothed = 0.0;
    m_slope_torque_current = 20.0;
    // ----------------------------------------------
```

### Bug 2: REST API Steering Range Carry-Over

**The Issue:**
If a car has an encrypted/broken steering range in shared memory, the app asks the `RestApiProvider` to fetch it via HTTP. The result is stored in `m_fallbackRangeDeg`.
If you switch from a car with a 540° lock to a car with a 900° lock (and both are encrypted), the app will continue using the old 540° lock while it waits for the new HTTP request to finish, causing the Gyro Damping and Soft Lock to calculate incorrectly for the first few seconds.

**The Fix:**
1. In `src/io/RestApiProvider.h`, add a reset method:
```cpp
    void ResetSteeringRange() {
        m_fallbackRangeDeg.store(0.0f);
    }
```
2. In `src/ffb/FFBMetadataManager.cpp`, call it when a car change is detected:
```cpp
    if (vehicleName && std::strcmp(vehicleName, m_vehicle_name) != 0) {
        StringUtils::SafeCopy(m_vehicle_name, STR_BUF_64, vehicleName);
        changed = true;

        RestApiProvider::Get().ResetManufacturer();
        RestApiProvider::Get().ResetSteeringRange(); // <-- NEW FIX
        RestApiProvider::Get().RequestManufacturer(6397, m_vehicle_name);
    }
```

### Bug 3: Teleport Derivative Spikes (The "Garage to Track" Jolt)

**The Issue:**
In `FFBEngine.cpp`, you track the previous frame's data to calculate velocities (e.g., `m_prev_vert_deflection`, `m_prev_susp_force`, `m_prev_rotation`).
When you click "Drive", the game teleports the car from the garage to the track. The telemetry values jump discontinuously.
For example, `(track_deflection - garage_deflection) / dt` results in a suspension velocity of thousands of meters per second for exactly 1 frame.
Currently, the `FFBSafetyMonitor` catches this and clamps the physical output to the wheel, but the math still spikes, which pollutes your smoothing filters.

**The Fix:**
We need a flag to re-seed the derivative variables whenever the car transitions from an "unallowed" state (Garage/Menu) to an "allowed" state (Driving).

1. In `src/ffb/FFBEngine.h`, add a flag:
```cpp
    bool m_derivatives_seeded = false;
```

2. In `src/ffb/FFBEngine.cpp` inside `calculate_force`, find the transition block and set the flag to false:
```cpp
    if (m_was_allowed && !allowed) {
        m_upsample_shaft_torque.Reset();
        // ... existing resets ...
        m_derivatives_seeded = false; // <-- NEW FIX
    }
```

3. Further down in `calculate_force`, right before you start calculating effects (around line 250), re-seed the variables if needed:
```cpp
    // --- NEW FIX: Re-seed derivatives after teleport/garage ---
    if (!m_derivatives_seeded) {
        for (int i = 0; i < 4; i++) {
            m_prev_vert_deflection[i] = upsampled_data->mWheel[i].mVerticalTireDeflection;
            m_prev_rotation[i] = upsampled_data->mWheel[i].mRotation;
            m_prev_brake_pressure[i] = upsampled_data->mWheel[i].mBrakePressure;
            m_prev_load[i] = upsampled_data->mWheel[i].mTireLoad;
            m_prev_slip_angle[i] = 0.0; // Reset LPF state
        }
        m_prev_susp_force[0] = upsampled_data->mWheel[0].mSuspForce;
        m_prev_susp_force[1] = upsampled_data->mWheel[1].mSuspForce;
        m_prev_vert_accel = upsampled_data->mLocalAccel.y;

        float range = upsampled_data->mPhysicalSteeringWheelRange > 0 ? upsampled_data->mPhysicalSteeringWheelRange : DEFAULT_STEERING_RANGE_RAD;
        m_prev_steering_angle = upsampled_data->mUnfilteredSteering * (range / 2.0);

        m_derivatives_seeded = true;
    }
```

### Bug 4: Incomplete Normalization Reset

**The Issue:**
In `FFBEngine::ResetNormalization()`, you correctly reset `m_static_front_load`, but you forgot to reset `m_static_rear_load` and `m_last_raw_torque`.
If the new car spawns with a high torque value, `m_last_raw_torque` (holding the old car's value) will cause `torque_slew` to spike, which falsely triggers the Contextual Spike Rejection logic.

**The Fix:**
In `src/ffb/FFBEngine.cpp`, update `ResetNormalization()`:

```cpp
void FFBEngine::ResetNormalization() {
    std::lock_guard<std::recursive_mutex> lock(g_engine_mutex);

    m_metadata.ResetWarnings();

    m_session_peak_torque = (std::max)(1.0, (double)m_target_rim_nm);
    m_smoothed_structural_mult = 1.0 / (m_session_peak_torque + EPSILON_DIV);
    m_rolling_average_torque = m_session_peak_torque;
    m_last_raw_torque = 0.0; // <-- NEW FIX

    ParsedVehicleClass vclass = ParseVehicleClass(m_metadata.GetCurrentClassName(), m_metadata.GetVehicleName());
    m_auto_peak_front_load = GetDefaultLoadForClass(vclass);

    m_static_front_load = m_auto_peak_front_load * 0.5;
    m_static_rear_load = m_auto_peak_front_load * 0.5; // <-- NEW FIX
    m_static_load_latched = false;

    // ... rest of the function
```

## Summary

By applying these 4 fixes alongside the missing telemetry counter resets we discussed earlier, you guarantee that the physics engine starts with a 100% clean slate every time the user switches cars or leaves the garage. This will completely eliminate random FFB jolts on spawn and ensure the math behaves predictably.
